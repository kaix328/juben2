# 📋 文档管理规范

## 🎯 核心原则

### 1. 定期清理过时文档
- ⏰ **频率**: 每周或每个重要里程碑后
- 🗑️ **清理对象**: 
  - 重复的完成报告
  - 过时的临时文档
  - 已被替代的旧版本文档
- ✅ **保留标准**: 
  - 是否是最新版本
  - 是否有独特价值
  - 是否被其他文档引用

### 2. 避免创建重复的完成报告
- ❌ **不要**: 每次完成任务都创建新的完成报告
- ✅ **应该**: 更新主要的总结文档
- 📝 **例外**: 重大里程碑可以创建独立报告

### 3. 保持文档结构简洁
- 📁 **根目录**: 只保留核心文档（<25个）
- 📂 **子目录**: 按类型分类（如果需要）
- 🏷️ **命名**: 清晰、一致、易于理解

### 4. 更新主要文档而不是创建新文档
- 📄 **主要文档**: 
  - `README.md` - 项目概览
  - `OPTIMIZATION_SUMMARY.md` - 优化总结
  - `CHANGELOG.md` - 变更日志
- ✏️ **更新方式**: 在现有文档中添加新章节
- 🆕 **创建新文档**: 仅当内容完全不同或文档过长时

---

## 📚 文档分类标准

### 必须保留的文档
1. **项目主文档**
   - README.md
   - CHANGELOG.md
   - ATTRIBUTIONS.md

2. **主要总结文档**
   - OPTIMIZATION_SUMMARY.md（持续更新）
   - 最新的完成报告（1个）

3. **配置指南**
   - ENV_SETUP.md
   - DOCKER_DEPLOYMENT.md
   - 其他配置指南

4. **技术文档**
   - 架构设计
   - API文档
   - 部署指南

### 可以删除的文档
1. **重复的完成报告**
   - 多个"FINAL_COMPLETE"报告
   - 多个"TODAY_SUMMARY"报告
   - 多个"TASKS_DONE"报告

2. **过时的临时文档**
   - 旧的进度报告
   - 已完成的计划文档
   - 临时通知

3. **被替代的文档**
   - 旧版本的指南
   - 已合并到主文档的内容

## 🔄 文档更新流程

### 日常更新
```
1. 完成任务
2. 更新 OPTIMIZATION_SUMMARY.md
3. 更新 CHANGELOG.md
4. 不创建新的完成报告
```

### 重大里程碑
```
1. 完成重大功能
2. 更新 OPTIMIZATION_SUMMARY.md
3. 创建独立的完成报告（如果必要）
4. 更新 README.md
5. 清理过时文档
```

### 每周清理
```
1. 检查所有文档
2. 删除重复内容
3. 合并相似文档
4. 更新文档索引
```

## 📝 文档命名规范

### 推荐的命名
- ✅ `README.md` - 清晰明了
- ✅ `OPTIMIZATION_SUMMARY.md` - 描述性强
- ✅ `ENV_SETUP.md` - 简洁明确

### 避免的命名
- ❌ `FINAL_FINAL_COMPLETE.md` - 重复词汇
- ❌ `TODAY_SUMMARY_2026_01_19.md` - 日期在文件名中
- ❌ `TEMP_DOC.md` - 临时文档应及时删除

## 🎯 文档数量控制

### 目标
- **根目录**: <25个Markdown文件
- **总文档**: <50个Markdown文件
- **清理频率**: 每周或每个里程碑

### 警戒线
- ⚠️ **根目录>30个**: 需要清理
- 🚨 **根目录>50个**: 立即清理
- 🔴 **总文档>100个**: 严重问题
